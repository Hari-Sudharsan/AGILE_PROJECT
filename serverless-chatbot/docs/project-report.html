1.2 Architecture Explanation

The system follows a serverless microservice pattern. The React frontend is hosted as static files on S3. All API calls go through API Gateway, which triggers a Lambda function. Lambda runs our FastAPI backend using the Mangum adapter (which translates Lambda events into ASGI requests).

DynamoDB stores structured chat logs. CloudWatch automatically collects Lambda logs for monitoring. The ML layer runs as separate scripts (locally or via scheduled Lambda) to simulate MLOps workflows.

3.1 Backend — FastAPI Chat Endpoint

backend/main.py Python 
# Request model — Pydantic validates input automatically
class ChatRequest(BaseModel):
    message: str
    session_id: str = "default"

    @validator("message")
    def message_must_not_be_empty(cls, v):
        v = v.strip()
        if not v:
            raise ValueError("Message cannot be empty")
        if len(v) > 500:
            raise ValueError("Message too long (max 500 chars)")
        return v

@app.post("/chat", response_model=ChatResponse)
def chat(request: ChatRequest):
    start_time = time.time()
    # 1. Log request   2. Get bot reply   3. Save to DB
    log_request(request.session_id, request.message)
    bot_reply = get_bot_response(request.message)
    elapsed_ms = round((time.time() - start_time) * 1000, 2)
    save_chat_log(request.session_id, request.message,
                  bot_reply, timestamp, model_version)
    track_response_time(elapsed_ms)
    return ChatResponse(response=bot_reply, ...)

3.2 Chat Handler — Rule-Based + LLM fallback

backend/chat_handler.py Python 
# Pattern-response pairs using regular expressions
RESPONSE_RULES = [
    (r"\b(hi|hello|hey)\b",
     ["Hello! How can I help you? 😊",
      "Hi there! What's on your mind?"]),
    (r"\b(bye|goodbye)\b",
     ["Goodbye! Have a great day! 👋"]),
    # ... more rules ...
]

def get_bot_response(user_message: str) -> str:
    # Priority 1: math expressions
    math_result = try_math_eval(user_message)
    if math_result: return math_result

    # Priority 2: rule-based matching
    for pattern, responses in RESPONSE_RULES:
        if re.search(pattern, user_message.lower()):
            return random.choice(responses)

    # Priority 3: OpenAI API (if key set in .env)
    if os.getenv("OPENAI_API_KEY"):
        return call_openai_api(user_message)

    # Priority 4: fallback
    return "I'm still learning! Try asking something else."

3.3 Database — SQLite + DynamoDB Dual Support

def save_chat_log(session_id, user_message,
                    bot_response, timestamp, model_version):

Section 04

🧠 MLOps Implementation

📊
Data Collection
Every user message and bot response is saved to chat_logs.json. This becomes the training dataset for future model improvements.
🏋️
Training Workflow
Run python ml/train.py to simulate a full pipeline: load data → analyse → train (5 epochs) → evaluate → version update.
🏷️
Model Versioning
ml/version.txt stores the current version. Every successful training run increments the PATCH number (e.g. 1.0.0 → 1.0.1).
📈
Monitoring
Response times and error counts are tracked in metrics.json. In production, monitor.py sends these to AWS CloudWatch.

CloudWatch Usage (Production)

In production, all Lambda print() and logging output is automatically captured by AWS CloudWatch Logs. You can then run Log Insights queries:

CloudWatch Log Insights QuerySQL
-- Find average response time in last hour
fields @timestamp, @message
| filter @message like "RESPONSE"
| parse @message '"response_time_ms": *,' as rt
| stats avg(rt) as avg_response_ms

-- Alert: count errors in last 5 minutes
fields @timestamp
| filter @message like "ERROR"
| stats count() as errors by bin(5m)

☁️ AWS Deployment — Step-by-Step

Pre-requisites: AWS account, AWS CLI installed (aws configure), Python 3.11, Node.js 18.

Step 1 — Create DynamoDB Table

  1. Go to AWS Console → DynamoDB → Create Table
  2. Table name: ChatLogs | Partition key: id (String)
  3. Leave all other settings as default and click Create

Step 2 — Package & Deploy Backend to Lambda

TerminalShell
# Package backend + dependencies into a ZIP
cd backend
pip install -r requirements.txt --target ./package
cp *.py ./package/
cp -r ../ml ./package/ml
cd package && zip -r ../../backend.zip . && cd ..

# Create the Lambda function (first time)
aws lambda create-function \
  --function-name serverless-chatbot \
  --runtime python3.11 \
  --handler lambda_handler.handler \
  --zip-file fileb://backend.zip \
  --role arn:aws:iam::YOUR_ACCOUNT_ID:role/lambda-role

# Update Lambda (subsequent deploys)
aws lambda update-function-code \
  --function-name serverless-chatbot \
  --zip-file fileb://backend.zip

Step 3 — Configure Environment Variables

TerminalShell
aws lambda update-function-configuration \
  --function-name serverless-chatbot \
  --environment "Variables={
    ENVIRONMENT=production,
    DB_TABLE_NAME=ChatLogs,
    AWS_REGION=us-east-1,
    MODEL_VERSION=1.0.0
  }"

Step 4 — Create API Gateway

  1. Go to API Gateway → Create API → REST API → Build
  2. Click Actions → Create Resource, name it chat (path: /chat)
  3. Select the /chat resource → Actions → Create Method → POST
  4. Integration type: Lambda Function → select serverless-chatbot
  5. Enable CORS: Actions → Enable CORS → Yes, replace existing CORS headers
  6. Actions → Deploy API → New Stage: prod → Copy the Invoke URL

Step 5 — Host Frontend on S3

TerminalShell
# 1. Set API URL in frontend
echo "REACT_APP_API_URL=https://YOUR-API-ID.execute-api.us-east-1.amazonaws.com/prod" \
  > frontend/.env.production

# 2. Build React app
cd frontend && npm run build

# 3. Create S3 bucket (name must be globally unique)
aws s3 mb s3://my-chatbot-frontend-2024

# 4. Upload build files
aws s3 sync build/ s3://my-chatbot-frontend-2024 --acl public-read

# 5. Enable static website hosting
aws s3 website s3://my-chatbot-frontend-2024 \
  --index-document index.html

GitHub Actions YAML (key sections)

.github/workflows/ci-cd.ymlYAML
name: 🤖 Chatbot CI/CD Pipeline

on:
  push:
    branches: [ main, develop ]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Install deps
        run: pip install -r backend/requirements.txt
      - name: Run tests
        run: pytest tests/ -v --tb=short

  build-frontend:
    needs: test          # Only runs if tests pass
    runs-on: ubuntu-latest
    steps:
      - run: cd frontend && npm ci && npm run build

  deploy:
    needs: [test, build-frontend]
    if: github.ref == 'refs/heads/main'   # main branch only
    steps:
      - # Configure AWS credentials from GitHub Secrets
      - run: aws lambda update-function-code --zip-file fileb://backend.zip
      - run: aws s3 sync frontend/build/ s3://${{ secrets.S3_BUCKET_NAME }}

Section 08

🔐 Security Measures

✅
Input Validation
Pydantic validators reject empty messages and anything over 500 characters. Invalid requests return 400/422 before reaching the chatbot logic.
🔑
Environment Variables
API keys (OpenAI, AWS) are stored in .env files, never in code. .env is gitignored. GitHub Secrets used in CI/CD.
🚫
No eval() on User Input
The math evaluator uses regex to extract only valid number+operator patterns before evaluating — prevents code injection.
🌐
CORS Configuration
FastAPI CORS middleware restricts which domains can call the API. In production, replace allow_origins=["*"] with your specific frontend URL.

Future Security Improvements

For a production system, also consider: JWT-based user authentication, AWS WAF (firewall) in front of API Gateway, rate limiting per user/IP, input sanitization against XSS/injection, and HTTPS-only enforcement.
